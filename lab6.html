For the prelab, I set up a debugging system that uses Bluetooth to send ToF data from the Artemis to my laptop. Therefore, the strategy I used was to store all of the data on the Artemis’ memory and send it all at once after the task was over.

P Controller

I first chose to implement a P (proportional) controller. To implement a P controller, I first set a target distance (a.k.a. reference input in control theory) of 300 mm. That is, the goal is to have the robot stop at a distance 300 mm from the wall. The error is defined as the current distance minus the target distance. And to reach the target distance as fast as possible, the input is made to be proportional to the error — that is, the larger the error, the larger the input. PD Controller

Adding a derivative term has the effect of preventing an overshoot by easing up the control input as the robot approaches the target distance. To implement a PD controller, I had to modify the code so that it keeps track of the error from the previous loop. Then I calculated the change in error by computing the current error minus the previous error. I scaled this difference by a constant Kd and added this to the proportional term. Note: I realized during the implementation of the PD controller that each control loop took anywhere between 0.2 second to 1 second which seemed way too long. It turned out that calling stopRanging() and startRanging() every loop caused each loop to take a long time. To fix this, I just called startRanging() once before the control loop and never stopped taking measurements during the control loop. This dropped the time taken for each loop to about 0.01 second. Running just a P controller with this loop frequency would probably be enough to reach the target distance.
